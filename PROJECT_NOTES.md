# RouteWolf — заметки для передачи контекста

Файл для того, чтобы продолжить работу над проектом без потери контекста —
неважно, здесь или на другой площадке/с другой моделью. Обновляется по ходу
работы, актуален на v52. Читать перед следующими изменениями в
`routewolf-install.sh`.

## Что это за проект

Форк `domain-routing-openwrt` для OpenWrt. Выборочная маршрутизация трафика
через VPN-туннель по спискам доменов/IP, а не всего интернета целиком.

Механизм: `dnsmasq (nftset=) → набор nftables vpn_domains → fwmark 0x1 →
ip rule → table vpn → default через туннель`. IPv4 CIDR идут отдельным
статическим набором `vpn_subnets` через `loadfile`.

Поддерживаемые туннели: WireGuard, AmneziaWG, OpenVPN, Sing-box (VLESS
Reality), Outline. Ставится на OpenWrt 24.10 (opkg) и 25.12+ (apk), задача —
также ImmortalWrt/X-WRT.

Основной файл — `routewolf-install.sh`, один большой POSIX-sh скрипт
(~5600 строк на v52). Это одновременно и установщик, и источник heredoc-ов,
которые пишут остальные скрипты на роутер.

## САМОЕ ВАЖНОЕ: дублирование функций между установщиком и рантаймом

Это будет ломать любые будущие правки, если не держать в голове.

`routewolf-install.sh` в процессе установки пишет на роутер отдельные,
самостоятельные скрипты через `cat << 'EOF' > /путь/файл ... EOF`
(кавычки вокруг `EOF` = без подстановки переменных, буквальный текст).
Крупнейшие:

- `/usr/sbin/rw` (heredoc строки ~3869–4341 на v52) — вся командная оболочка
  `rw help/status/dco/lists/...`
- `/usr/sbin/routewolf-route.sh` — определение шлюза и установка маршрута
  в таблицу vpn
- `/usr/sbin/routewolf-update.sh`, `/usr/sbin/routewolf-apply.sh` — обвязка
  вокруг обновлений
- `/etc/init.d/routewolf` — init-скрипт, его `start()` перезагружает списки

Текст внутри heredoc **не выполняется установщиком**. Это данные для `cat`,
которые оседают в отдельном файле и запускаются позже, отдельным процессом,
когда пользователь наберёт `rw ...` или сработает cron. Если внутри такого
heredoc текстуально написано `some_function() { ... }`, это НЕ определяет
функцию в самом установщике — установщик её вызвать не может.

**Настоящий баг v51**, попавший даже в протестированный код: DCO-логика
(`install_dco_if_possible`, `dco_kmod_loaded`) была добавлена только внутри
heredoc `/usr/sbin/rw`. Установщик (`ovpn_harden_route_only_config`,
реальная функция самого себя, ВНЕ heredoc) пытался её вызвать — получал
`command not found`, молча проваливался в `else`. Синтаксис (`sh -n`)
это не ловит — файл валиден, ошибка чисто в namespace, всплывает только
при реальном запуске. Тест в песочнице тоже сначала не поймал баг — я
извлекал функции по имени через `awk`, что при дублировании имени в двух
местах давало ложное срабатывание "работает".

**Практическое следствие**: если функция нужна и установщику (во время
настройки), и рантайм-инструменту `rw` (после установки) — заводить ДВЕ
копии, одну как обычную top-level функцию установщика (ДО того места, где
она вызывается), вторую — внутри нужного heredoc. Не полагаться на то, что
одна и та же функция "видна" в обоих местах. По этой же причине в четырёх
местах (не в одном) существует фикс awk для определения шлюза — исторически
тот же паттерн дублирования, тот же класс бага, только найден раньше.

**Как проверять перед тем как звать функцию X из места Y**: `grep -n
"^X() {" routewolf-install.sh`, посмотреть на номера строк, сравнить с
границами ближайшего heredoc (`awk '/cat << .?EOF.? >/{print NR": "$0}'`,
затем найти закрывающий `EOF` на отдельной строке).

## Три канала обновления — не путать

| Канал | Кто ходит | Что тянет | Когда трогает код |
|---|---|---|---|
| **Gen A** (`getdomains`, самые старые установки) | легаси-пользователи | только `lists/*` | никогда |
| **Gen B / Legacy** (`update.sh`) | текущие установки без переустановки | либо только списки (`--auto`, cron 02:00), либо полный тарбол+установщик (без `--auto`, `rw update`) | только при ручном/не-auto вызове |
| **Интерактивный установщик** | явный повторный запуск `install.sh`/`routewolf-install.sh` | всё | всегда, с меню выбора |

`update.sh` — единственный файл с ДВУМЯ режимами в одном:
- вызван с `--auto` (только cron) → лёгкий агент, сравнивает хеш списков,
  ничего не делает, если не изменилось, никогда не трогает код;
- вызван без `--auto` (человек, `rw update`) → оригинальный bootstrapper
  без изменений, качает тарбол, зовёт `routewolf-install.sh --update`.

`routewolf-update.sh` (генерируется, стоит на роутере) просто форвардит
`"$@"` в `update.sh` — cron даёт `--auto`, `rw update` не даёт ничего.

**`update_existing_installation()` (обрабатывает `--update`) НЕ вызывает
`ovpn_harden_route_only_config`** — то есть даже полное обновление кода не
трогает уже вставленный `.ovpn` и не включит/выключит DCO на существующем
туннеле. Это осознанное решение: контент с ключами не должен переписываться
автоматически. `install_management_commands()` (переписывает `/usr/sbin/rw`
и прочие рантайм-скрипты) `--update` вызывает — значит после апдейта сама
команда `rw dco on` становится актуальной, а флаг в файле — нет.

## Повторная установка поверх существующей — `detect_existing_routing_config()`

При запуске установщика (не `--update`, а обычный прогон) он ищет следы уже
настроенного тоннеля/маршрутизации (UCI-секции, `/etc/routewolf/route.conf`
и т.д.) и, если находит, показывает меню:

1. **Обновить код/списки/команды, VPN-конфиг не трогать** (по умолчанию).
   Даже здесь `route_vpn()` вызывается и **перезаписывает**
   `/usr/sbin/routewolf-route.sh` — то есть баг-фиксы в логике определения
   маршрута доедут даже по этому "безопасному" пути. DCO/`.ovpn` — нет.
2. **Заменить старый конфиг**. Зовёт `cleanup_existing_routing_config()`
   (сносит UCI-секции туннеля/файрвола, ключи придётся вставлять заново),
   потом идёт по полному интерактивному потоку настройки — тут
   `ovpn_harden_route_only_config` вызовется, DCO получит honest-проверку.
3. Диагностика, без изменений.

## Маркер версии (`v52`)

`/etc/routewolf/installed_version` — что реально стоит на роутере сейчас,
пишется в конце ЛЮБОГО успешного пути (`--update`, пункт 1, пункт 2, чистая
установка). Сравнивается с `$PROJECT_VERSION` установщика, который был
запущен, и показывается в меню `detect_existing_routing_config`, чтобы
человек видел разницу версий вместо гадания вслепую.

НЕ путать с корневым файлом `version` в репозитории — тот, по задумке из
v49, для будущего канала v2 (ещё не реализован по факту, не читается сейчас
никаким кодом), сравнение по хешу содержимого списков в `update.sh --auto`
устроено отдельно и не завязано на этот файл.

## Известные баги, найденные и исправленные (хронологически)

1. **awk определение шлюза OpenVPN** — `split()` даёт строку `"0"`, `awk`
   трактует её как false в `if (o[1] && o[2] && o[3])`. Ломало любой адрес
   туннеля с нулём в 3-м октете — в том числе `10.8.0.0/24`, дефолт самого
   OpenVPN. Фикс: `!= ""` вместо проверки на истинность. 4 места (см. раздел
   про дублирование).
2. **Fail-open вычищал рабочий маршрут**. Не найдя шлюз, код удалял
   `default` из таблицы `vpn` вместо безшлюзового `dev tunX scope link`,
   который уже был разрешён для wg/awg и прекрасно работает на
   point-to-point интерфейсах (NOARP). Фикс: если устройство
   `POINTOPOINT` — безшлюзовый маршрут, не пустая таблица.
3. **`disable-dco` жёстко всегда включён**. Автор перестраховался из-за
   реального риска (endless-reconnect на части сборок). Фикс: включать
   DCO только если `modprobe` + `lsmod` подтвердили загрузку модуля —
   тот же принцип, что нужен и для AmneziaWG на ImmortalWrt/X-WRT
   (см. TODO ниже, там ещё не сделано).
4. **Namespace-баг из-за heredoc-дублирования** (см. раздел выше) —
   установщик звал функцию, которая существовала только в чужом heredoc.
5. **`dco_status` сам себе противоречил** — показывал "модуль загружен,
   DCO активен" одновременно с "disable-dco присутствует", хотя второе
   всегда должно перекрывать первое. Фикс: одна честная строка, сводящая
   оба факта, вместо двух независимых.
6. **DCO ломается во время работы, не при загрузке** — verified-loaded
   (`modprobe`+`lsmod`) недостаточно. Живая улика:
   `dmesg: ovpn_encrypt_one: error while retrieving primary key slot`,
   после которой tun0 теряет IPv4-адрес без явного `ifdown`. Модуль
   `kmod-ovpn-dco-v2 6.6.141.0.2.20251017-r1`, сервер AntiZapret. Фикс:
   дефолт при установке откачен на `disable-dco` безусловно (v53); проверка
   через modprobe/lsmod оставлена только для `rw dco on/off/status`, как
   осознанный ручной opt-in с предупреждением о точной сигнатуре в логе.

**Обновление**: найдена вероятная первопричина в апстриме. GitHub-issue
`OpenVPN/ovpn-dco#95` — тот же класс отказа (DCO рвёт сессию около момента
`reneg-sec`, ключ теряется при ротации). Причина установлена авторами:
рассинхронизация типов u8/u16 при чтении key-id, фикс в PR
`OpenVPN/ovpn-dco#94`, подтверждён репортёром бага. Патч описан как
затрагивающий **big-endian** платформы конкретно — наша строка ошибки
чуть другая (`ovpn_encrypt_one` вместо `ovpn_decrypt_one`), так что это
либо тот же баг, либо близкий родственник в том же участке кода. Пакет
`kmod-ovpn-dco-v2` на роутере пользователя датирован октябрём 2025 —
раньше обсуждения фикса (февраль 2026), однозначно без него. В master-ветке
`openwrt/packages` уже есть пересборка под новым именем `ovpn-backports`
(вместо `kmod-ovpn-dco-v2`), версия датирована мартом 2026 — вероятно
содержит фикс, но не подтверждено, дошла ли до стабильного фида 24.10.
Дальнейший шаг — не наш код, а `opkg update && opkg list kmod-ovpn-dco-v2`
на роутере, ждать когда фид подтянет обновлённый пакет.

**v54**: дефолт из v53 (безусловный `disable-dco`) заменён интерактивным
выбором при установке — `ask_dco_choice()`, использует ранее неиспользуемый
`ask_yes_no_global()`. Причина: находка про DCO была с одного роутера, у
других связка работает нормально, жёсткий дефолт "всегда выкл" был бы
избыточно консервативен. Выбор пользователя — ОДНО из двух условий в
`ovpn_harden_route_only_config`; verified-check через `install_dco_if_possible`
остаётся вторым, обязательным условием даже при явном "да".

## Инцидент DCO — закрыт, вывод изменил дефолт (v53)

Пользователь включил DCO (`rw dco on`), туннель отвалился. Диагностика дала
конкретную улику, не просто корреляцию:

```
dmesg: [24737.494778] ovpn_encrypt_one: error while retrieving primary key slot
ip addr show tun0: только inet6 link-local, IPv4-адреса нет
```

При этом в `logread` чуть раньше — успешное поднятие: TLS handshake прошёл,
`net_addr_v4_add: 10.28.0.4/22 dev tun0`, `Initialization Sequence Completed`.
То есть модуль ядра `kmod-ovpn-dco-v2 6.6.141.0.2.20251017-r1` не просто
"не грузится" (это уже проверялось через `modprobe`/`lsmod` и подтверждалось
как ОК) — он ломается **во время работы**, при ротации ключа шифрования.
`modprobe`/`lsmod` в принципе не может поймать этот класс отказа: они видят
"модуль в памяти", а не "модуль переживёт rekey через несколько часов".

**Вывод**: verified-loaded (v51-v52) — необходимое, но недостаточное условие.
Дефолт при установке **откачен обратно на `disable-dco`** (v53), независимо
от результата `install_dco_if_possible`. Сама функция и связанная с ней
инфраструктура (`dco_kmod_loaded`, `rw dco on/off/status`) не удалена —
она по-прежнему нужна для отдельной, реальной проблемы (пакет ставится,
модуль вообще не грузится, актуально для ImmortalWrt/X-WRT), и остаётся
доступной как осознанный ручной opt-in.

`rw dco on` и `rw dco status` теперь сами печатают точную сигнатуру этого
бага (`dmesg | grep 'primary key slot'`), чтобы при повторении не проходить
диагностику с нуля.

**Не сделано, важный TODO**: самовосстановление в watchdog — если DCO включён
и tun0 систематически теряет IPv4 через N проверок, автоматически откатывать
на `rw dco off` и перезапускать. Требует времени на реальном роутере для
проверки (нужно дождаться повторного естественного возникновения бага, не
проверяется мгновенно в песочнице).

## Открытые задачи (обсуждались, не реализованы)

- **Watchdog self-healing для DCO** — см. раздел "Инцидент DCO" выше:
  автооткат на `rw dco off`, если tun0 систематически теряет IPv4 при
  включённом DCO. Не реализовано, нужна проверка на реальном роутере во
  времени.
- **`rw ipv6 on/off`** — не тривиально: секции ipv6 в файрволе создаются
  только внутри `update_existing_installation()`, не в рантайм-скриптах.
  Просто поправить `user.conf` недостаточно.
- **`rw uninstall`** — обёртка над существующим `uninstall.sh`, нужно решить
  про подтверждение и легаси-слой `getdomains`.
- **Переключение протокола на лету** — `WATCHDOG_TYPE` сейчас выставляется
  один раз при установке, живёт в `watchdog.conf`. Живое переключение
  затронет несколько файлов разом.
- **LuCI-интерфейс** — упирается в то, что конфиг хранится в shell-файлах
  (`/etc/routewolf/user.conf`, простые `VAR='value'`), а не в UCI
  (`/etc/config/routewolf`). Штатный способ (CBI) работает поверх UCI.
  Нужна миграция формата конфига, прежде чем LuCI-страница напишется
  "сама". Альтернатива — руками читать/писать shell-файлы из Lua, более
  хрупко.
- **AmneziaWG на ImmortalWrt/X-WRT** — `install_awg_packages` уходит сразу
  во внешний фид `Slava-Shchipunov/awg-openwrt`, собранный под официальные
  ядра OpenWrt. На форках другой vermagic — пакет ставится, модуль не
  грузится, `modprobe amneziawg` нигде не проверяется (0 вхождений).
  Нужен тот же паттерн, что уже сделан для OpenVPN DCO: сначала штатный
  репозиторий системы, потом внешний фид как fallback, обязательная
  проверка `modprobe`.
- **Определение opkg/apk** — сейчас по наличию бинарника
  (`command -v apk` первым), путает 24.10/25.12 на переходных сборках.
  Надо смотреть на живую базу пакетов (`/etc/apk/world` vs
  `/usr/lib/opkg/status`).
- **OpenVPN: `route-noexec` + `route-up`** — вместо вычисления шлюза как
  догадки (`x.y.z.1`) брать настоящий шлюз, который сервер отдаёт через
  `$route_vpn_gateway`. Пока сделан только безопасный fallback
  (безшлюзовый маршрут для point-to-point), не полная реализация идеи.
- **AWG-параметры v1/v1.5/v2** — `awg_s3/s4/i1..i5` пишутся в UCI без
  проверки, понимает ли их установленный kmod. Надо сверяться с тем, что
  реально объявляет `/lib/netifd/proto/amneziawg.sh`.

## Как эта сессия тестирует изменения (следовать этому же подходу)

1. Изменения в код — через `str_replace` с точным `old_str`/`new_str`,
   не переписыванием файлов целиком.
2. `sh -n routewolf-install.sh` после каждой правки — ловит синтаксис,
   НЕ ловит namespace-баги (см. раздел про heredoc).
3. Изолированный тест: вырезать ровно нужный диапазон строк (проверить
   границы функции через `awk`, не полагаться на память о номерах строк —
   они сдвигаются после каждой правки), подставить моки (`opkg`/`modprobe`/
   `lsmod`/`find_ovpn_config` и т.д. через PATH или sourced stubs), прогнать
   оба сценария (успех/неудача), проверить фактический вывод, не только код
   возврата.
4. Диффом от чистого `RouteWolf-main` (нетронутая v48-копия, лежит в
   `/home/claude/RouteWolf-main`) проверять, что заморожённые файлы
   (`install.sh`, `getdomains-install.sh`, `uninstall.sh`, `cleanup.sh`,
   `routewolf-check.sh`, `lists/ipv4.lst`, `lists/ipv6.lst`) не менялись —
   их трогать нельзя, легаси-установки зависят от их URL/формата как от
   публичного API.
5. Финальная проверка на реальном роутере пользователя — обязательна перед
   тем, как считать фикс подтверждённым. Несколько раз именно живой вывод
   `logread`/`ip route`/`nft list` ловил то, что песочница не могла
   (например, что netifd, а не OpenVPN, дёргал интерфейс каждые ~40 секунд —
   это не было предположением, обнаружено по временным меткам в реальном
   логе).

## Контакт с пользователем — стиль работы

Роутеры пользователя: Xiaomi 4C (слабый, 16 МБ flash / 64 МБ RAM — тестовый
для лимитов) и Xiaomi AX3000T (мощный, эталон "всё ок"). Целится в первую
очередь на мощные роутеры для новых клиентов; слабые — для тех, у кого уже
есть железо и его надо перепрошить. Использует в основном AmneziaWG,
IP-адресов в списках почти нет (Telegram — крайний случай). OpenVPN многие
клиенты просят, но с ним были проблемы с маршрутизацией (см. баги 1-2 выше).
VPN-сервер в текущих тестах — AntiZapret (`fin.amored.ru`).

Принципиально важно для пользователя: старые версии проекта у существующих
пользователей не должны сломаться ни при каком изменении. Отсюда — заморозка
файлов легаси-API, отдельные каналы обновления, маркер версии вместо
автоматического слепого апгрейда.
