# RouteWolf v54 — DCO: выбор при установке, не жёсткий дефолт

Включает всё из v49-v53 плюс интерактивный выбор DCO вместо безусловного
`disable-dco` из v53.

## Почему снова меняю

v53 сделала DCO безусловно выключенным при установке — правильно на момент
находки (живой баг ротации ключа, см. `CHANGES-v53.md` и найденный upstream
issue `OpenVPN/ovpn-dco#95`). Но это была информация с **одного** роутера.
У других связка OpenWrt + OpenVPN + AntiZapret с DCO работает нормально —
жёсткое «всегда выключено» для них означало бы terять скорость без причины.

## Что добавлено

Новая функция `ask_dco_choice()` — использует уже существовавший в проекте,
но нигде не применявшийся `ask_yes_no_global()`. Спрашивает один раз при
настройке OpenVPN (из вставленного `.ovpn` или при обнаружении уже
настроенного вручную), с честным объяснением компромисса и точной сигнатурой
известного бага (`dmesg | grep 'primary key slot'`), по умолчанию — «нет».

```
OpenVPN DCO переносит шифрование в ядро - быстрее, чем путь по умолчанию.
Известная проблема: на части сборок kmod-ovpn-dco-v2 туннель тихо теряет IP
через несколько часов работы (баг ротации ключа). У многих работает нормально,
у части сломано. Если сломается позже: dmesg | grep 'primary key slot', затем: rw dco off
Включить DCO (y/n) [n]:
```

Выбор пользователя (`DCO_USER_CHOICE`) идёт в `ovpn_harden_route_only_config()`
как ОДНО из двух условий. Даже явное «да» всё равно проходит через
`install_dco_if_possible()` (verified modprobe+lsmod) — согласие пользователя
не делает нерабочий модуль рабочим, это по-прежнему необходимое условие,
просто больше не единственное и не автоматическое.

## Проверено

Пять сценариев в изолированном тесте (`ovpn_harden_route_only_config`):
выбор off/on × модуль грузится/не грузится, плюс защитный дефолт при
неустановленной переменной — все дают ожидаемый результат, включая
безопасный откат, если пользователь выбрал «включить», а модуль всё равно
не подтвердился.

Отдельно проверен маппинг ответов в `ask_dco_choice()` через реальные (не
моки) `ask_yes_no_global`/`is_back`: `y`/`yes` → on, `n`/пусто/`назад` → off.

## Что НЕ менялось

`install.sh`, `getdomains-install.sh`, `uninstall.sh`, `cleanup.sh`,
`routewolf-check.sh`, `lists/ipv4.lst`, `lists/ipv6.lst` — не тронуты.

Существующие установки (в том числе те, что уже настроены на v53 с
`disable-dco`) этот выбор не увидят автоматически — вопрос задаётся только
при первичной настройке OpenVPN, тем же путём, что и раньше не трогал
уже вставленный `.ovpn`. Переключение на существующем туннеле — по-прежнему
`rw dco on`/`rw dco off`.
