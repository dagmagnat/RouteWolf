# RouteWolf v53 — DCO снова выключен по умолчанию (с уроком)

Включает всё из v49-v52 плюс откат дефолта DCO на основе реальной находки
на живом роутере пользователя.

## Что произошло

В v51 дефолт при установке был изменён: DCO включался, если `modprobe` +
`lsmod` подтверждали, что модуль ядра загрузился. Логика была рассчитана на
один конкретный класс отказа — пакет ставится, модуль не грузится
(vermagic mismatch на форках вроде ImmortalWrt/X-WRT).

На практике вскрылся **другой** класс отказа. Пользователь включил DCO,
туннель какое-то время работал нормально (TLS handshake прошёл, `tun0`
получил IPv4), а затем **сам потерял адрес**, без явного `ifdown`. В `dmesg`
нашлась точная причина:

```
ovpn_encrypt_one: error while retrieving primary key slot
```

Это ошибка ротации ключа шифрования **внутри самого модуля ядра**
(`kmod-ovpn-dco-v2 6.6.141.0.2.20251017-r1`, тестировалось с сервером
AntiZapret). `modprobe`/`lsmod` не может поймать такой отказ в принципе —
они проверяют «модуль загружен», а не «модуль переживёт rekey через
несколько часов работы». Это оказался ровно тот класс проблемы, о котором
в оригинале предупреждал автор комментарием про «endless reconnect на
части сборок» — просто проявляющийся глубже, чем можно было предположить
по одной лишь проверке загрузки.

## Фикс

Дефолт при установке — снова **безусловно `disable-dco`**, независимо от
результата `install_dco_if_possible`. Проверено в песочнице на обоих
сценариях (модуль грузится / не грузится) — теперь оба дают `disable-dco`.

Инфраструктура проверки (`dco_kmod_loaded`, `install_dco_if_possible`) **не
удалена** — она всё ещё нужна для `rw dco on/off/status` и всё ещё корректно
ловит отдельную, реальную проблему «пакет ставится, модуль вообще не
грузится». DCO остаётся доступен, но только как осознанный ручной шаг.

`rw dco on` и `rw dco status` теперь сами печатают точную сигнатуру этого
бага при включении DCO, чтобы при повторении не проходить диагностику с
нуля:

```
Known issue: some kmod-ovpn-dco-v2 builds can lose the tunnel's IPv4
address hours into a session, with this in the kernel log:
  dmesg | grep 'error while retrieving primary key slot'
```

## Открытый TODO

Watchdog не умеет автоматически откатывать DCO при повторении этого
симптома (tun0 теряет IPv4 через N проверок при включённом DCO). Идея
записана в `PROJECT_NOTES.md`, не реализована — требует времени на реальном
железе для проверки, естественное повторение бага не воспроизводится по
requesту.

## Проверено

- Синтаксис (`sh -n`) — OK.
- Изолированный тест `ovpn_harden_route_only_config`: оба сценария
  (`install_dco_if_possible` успех/неудача) теперь дают `disable-dco` в
  выходном конфиге — подтверждено, что откат безусловный, не зависит от
  состояния модуля.
- Живой источник данных для самого фикса — реальный `dmesg`/`logread`/
  `ip addr` с роутера пользователя, не предположение.

## Что НЕ менялось

`install.sh`, `getdomains-install.sh`, `uninstall.sh`, `cleanup.sh`,
`routewolf-check.sh`, `lists/ipv4.lst`, `lists/ipv6.lst` — не тронуты.
