## 1. Ночной cron больше не переустанавливает проект

**Было.** Каждую ночь в 02:00, независимо от того, изменилось ли хоть что-то:

- скачивался тарбол проекта (~600 КБ) в `/tmp` — а `/tmp` в OpenWrt это **tmpfs, то есть RAM**;
- проект полностью переустанавливался;
- перезаписывались скрипты в `/usr/sbin`, делался `uci commit` — **запись во flash каждую ночь**;
- выполнялось **2× `firewall restart` + 2× `dnsmasq restart`**.

На роутерах с 64 МБ ОЗУ и 16 МБ flash (Xiaomi 4C и подобные) это и есть источник
тормозов, зависаний туннеля и износа флеш-памяти. Списки обновлялись лишь как
побочный эффект этой переустановки.

**Стало.** `update.sh` при вызове с флагом `--auto` (так его зовёт только cron):

1. скачивает списки (~20 КБ) и сверяет контрольную сумму с кешем;
2. **не изменились → выходит, не тронув роутер вообще** — это ~95 % ночей;
3. изменились → отдаёт применение **родному коду роутера** (`/etc/init.d/routewolf start`);
4. тарбол не качает, проект не переустанавливает, во flash не пишет.

Ключевой принцип: новый код делает **только дешёвую проверку «изменилось ли»**.
Само применение списков выполняет старый, уже проверенный код с флешки роутера.
Новая логика в критический путь не добавляется.

## 2. Ручной запуск работает ровно как раньше

Лёгкий агент включается **только по флагу `--auto`**. Всё, что делает человек:

- команда обновления из README (`... update.sh | sh`),
- запуск из распакованного репозитория (`sh ./update.sh`),

идёт по **прежнему пути** — исходный bootstrapper сохранён в файле целиком и без
изменений. Ничего из документированного поведения не сломано.

## 3. Защита от сбоев (fail-open)

| Ситуация | Поведение |
|---|---|
| GitHub отдал HTML-страницу 404 | распознаётся, хеш **не** перезаписывается, откат к прежнему поведению |
| Закачка оборвалась (список похудел >2×) | не применяется, текущие списки сохраняются |
| Нет сети | откат к прежнему поведению |
| Списки отключены в `user.conf` | ничего не делается |
| Применение вернуло ошибку | хеш **не** запоминается, повтор следующей ночью |

Контрольная сумма записывается **только после успешного применения**. Поэтому
одна неудачная ночь не может навсегда заморозить роутер на битых списках.

Любая непредвиденная ошибка → поведение как в v48. Хуже, чем было, стать не может.

## 4. Список доменов: 670 → 589

`dnsmasq` с директивой `nftset=/youtube.com/` уже покрывает **все поддомены**.
Поэтому записи вроде `ads.youtube.com`, `accounts.google.com`, `clients1.google.com`
были избыточны — родитель уже в списке.

Удалена 81 такая запись. Покрытие проверено программно: **ни один домен не потерял
маршрутизацию**, новых записей не добавлено.

Это единственная оптимизация, которая доходит до самых старых установок —
они тянут с GitHub только списки и никогда не обновляют код.

## Как проверить перед публикацией

На одном своём роутере:

```sh
# положить новый update.sh во /tmp и прогнать дважды
sh /tmp/update.sh --auto     # 1-й раз: применит и запомнит контрольную сумму
sh /tmp/update.sh --auto     # 2-й раз: должен сказать "роутер не трогаю"
```

Если второй прогон пишет **«списки не изменились — роутер не трогаю»** — всё работает.

## Три канала обновления

| Поколение | Что тянет с GitHub | Получает код проекта |
|---|---|---|
| старые установки (`getdomains`) | только `lists/*` | никогда |
| текущие установки (RouteWolf) | `update.sh` + `lists/*` | **больше никогда** |
| v2 (в разработке) | `v2/*` по тегу | да, с проверкой версии |

Каталоги разделены физически: старые установки не знают про `v2/`, а v2 не смотрит
в старое. Роутер переходит на новый канал **только когда v2 установлен вручную**.
